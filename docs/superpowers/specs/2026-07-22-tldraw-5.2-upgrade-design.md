# tldraw 5.1.0 → 5.2.5 upgrade — design

- **Status:** SPEC (not started). Separate, self-contained piece of work.
- **Date:** 2026-07-22
- **Motivation:** Unblock clean wiring for the locked-shape affordance feature,
  and clear a small amount of version drift, at a cost that is now known to be
  low.
- **Companion docs:**
  - Trigger for this work: `2026-07-22-locked-shape-affordance-design.md` (in progress)
  - Strategic context: [`../../tldraw-replacement-analysis.md`](../../tldraw-replacement-analysis.md)

---

## 1. Why now — the trigger

The locked-shape affordance feature (a canvas-wide visual signal that a shape is
locked) wants to subclass tldraw's **`ShapeIndicatorOverlayUtil`**, mirroring the
pattern already established in `client/src/av/FadedCursorOverlay.ts`, which
subclasses `CollaboratorCursorOverlayUtil` and overrides
`render(ctx: CanvasRenderingContext2D, overlays)`.

On **5.1.0 that class is not part of `tldraw`'s public surface.** Verified in
`node_modules/tldraw/dist-cjs/index.d.ts`:

| line | what it shows |
|---|---|
| 61 | `import { ShapeIndicatorOverlayUtil } from '@tldraw/editor';` — imported for internal use |
| 1410 | appears inside the `defaultOverlayUtils` tuple type |
| 776 | `export declare class CollaboratorShapeIndicatorOverlayUtil` — the *collaborator* variant **is** exported |
| — | **no** `export declare class ShapeIndicatorOverlayUtil` — the local variant is **not** re-exported |

The app currently imports from `@tldraw/editor` in **zero** places, and
`@tldraw/editor` is **not a direct dependency** (transitive only). So on 5.1.0 the
feature would have to either add a pinned `@tldraw/editor` dependency purely as a
workaround, or avoid the idiomatic pattern.

**tldraw 5.2.0 moves `ShapeIndicatorOverlayUtil` from `@tldraw/editor` into
`tldraw`** — a listed breaking change that exists precisely to fix this. Upgrading
makes the feature's wiring clean and *removes* a dependency we would otherwise add.

> Note: 5.2.x adds **no** locked-shape UX of its own. This upgrade does not
> deliver the feature; it only makes the feature's wiring clean. The affordance is
> ours to build either way.

## 2. Scope — six packages, in lockstep

All tldraw-family packages must move together across **both** workspaces.

| package | workspace | current | target |
|---|---|---|---|
| `tldraw` | client | 5.1.0 | 5.2.5 |
| `@tldraw/sync` | client | 5.1.0 | 5.2.5 |
| `@tldraw/sync-core` | server | 5.1.0 | 5.2.5 |
| `@tldraw/tlschema` | server | 5.1.0 | 5.2.5 |
| `@tldraw/utils` | server | 5.1.0 | 5.2.5 |
| `@tldraw/validate` | server | 5.1.0 | 5.2.5 |

`@tldraw/sync` (client) and `@tldraw/sync-core` (server) are the two ends of the
canvas sync protocol. **They must be built and deployed together.**

## 3. Breaking-change impact

The five breaking changes listed for 5.2.0, each checked against the codebase:

| breaking change | impact | evidence |
|---|---|---|
| Triple/quadruple click events removed | **None** | Only a comment about xterm selection in `terminal/TerminalShapeUtil.tsx:273`; every other `triple` hit is unrelated `storage triple` naming in `server/src`. No tldraw triple-click handlers. |
| `ShapeIndicatorOverlayUtil` moved `@tldraw/editor` → `tldraw` | **Beneficial** | This is the reason for the upgrade (§1). Zero current `@tldraw/editor` imports, so nothing breaks; it only *enables*. |
| Minimum Node raised to `>=22.12.0` | **Satisfied** | Local `v22.23.1`; CI pins `node-version: 22.12.0` in `e2e.yml`, `canvas-v2-load.yml`, `canvas-v2-perf.yml`. No `engines` field to bump. |
| Collaborator APIs use branded `TLUserId` instead of `string` | **The only real work** — see §4 | Type-level only. |
| `StrokePoint.vector` / `setStrokePointRadii` removed | **None** | Zero usage anywhere in `client/src` or `packages/*/src`. |

**Caveat on provenance:** this table was derived from the published 5.2.0 release
notes. The implementer must read the full 5.2.0–5.2.5 changelogs before starting
and reconcile any change not listed here — treat §3 as a strong prior, not a
guarantee.

## 4. The only real work — branded `TLUserId`

Collaborator APIs now return/accept a branded `TLUserId` rather than a plain
`string`. This is a **compile-time** change: it surfaces as TypeScript errors
wherever a collaborator id is treated as a string.

The blast radius is contained, because the app **already funnels collaborator ids
through a single unwrap helper**, `rawUserId()` from `@ensembleworks/contracts`:

| call site | usage |
|---|---|
| `client/src/av/FadedCursorOverlay.ts:26` | `gains[rawUserId(userId)]` |
| `client/src/av/useSessionPulse.ts:70` | `rawUserId(identity)` |
| `client/src/av/leashes.tsx:42,44` | `rawUserId(peer.identity)`, `rawUserId(c.userId)` |
| `client/src/av/AvOverlay.tsx:11,24` | imports `rawUserId`; `editor.user.getId()` |

Also touching the boundary: `getDefaultUserPresence` (`client/src/App.tsx:9`) and
`editor.getCollaboratorsOnCurrentPage()` (`av/leashes.tsx:39`).

**Approach:** widen/adjust `rawUserId()`'s signature in `@ensembleworks/contracts`
to accept the branded type and return the plain string, so it remains the single
conversion boundary. Fix the handful of call sites that still assume `string`.
Resist scattering casts — one boundary, as today.

> This work sits in the A/V presence layer, which is the most incident-prone part
> of the app and the subject of open issues #54 and #55. That is an argument for
> doing it **in isolation**, not for avoiding it.

## 5. Risks

1. **Lockstep deploy (highest).** Client and server sync halves must ship
   together. A half-deployed upgrade breaks canvas sync on a live box. There is no
   partial-rollout path.
2. **Live production.** `ew-lsp-july` runs a live service (base always-on, surge
   hourly during lessons). Do **not** deploy this during a session.
3. **In-flight work collision.** At time of writing the working tree has
   uncommitted `client/src/canvas-health/` changes (another agent's
   connection-health modal work, see `docs/plans/2026-07-22-connection-health-modal-*`).
   A cross-cutting type change dropped underneath that will conflict.
4. **Type change in the A/V layer.** See §4 — contained, but it is the presence
   code path.
5. **Unit tests will not catch a sync-protocol regression.** Requires a real
   client↔server smoke (§6 step 4).

## 6. Plan

Own branch, nothing else on it.

1. Branch from a **clean** `main` (after the canvas-health work has landed — do
   not stack on it).
2. Bump all six packages (§2) to `5.2.5`. Install; confirm the lockfile resolves a
   single tldraw version (no duplicates).
3. Fix the `TLUserId` fallout (§4), boundary-first.
4. **Verify — all four, evidence before claiming done:**
   - client typecheck clean
   - full suite (`bun scripts/run-tests.ts`) — 213 suites currently green
   - **real sync smoke**: two browsers in one room, confirm shape edits propagate,
     presence/cursors render, and a terminal attaches. This is the check that
     matters and the one tests won't do.
   - locked shape still behaves as before (baseline for the follow-on feature)
5. Deploy client **and** server together, outside lesson hours.
6. Only then build the locked-shape affordance on top, importing
   `ShapeIndicatorOverlayUtil` from `tldraw`.

## 7. Rollback

Revert the branch and redeploy both halves together. No data migration is
involved and no persisted schema changes are expected — but confirm that last
point against the changelog for `@tldraw/tlschema` before deploying, since a
schema/migration change there *would* affect stored rooms and make rollback
non-trivial.

## 8. Relationship to the tldraw-replacement track

[`tldraw-replacement-analysis.md`](../../tldraw-replacement-analysis.md) (2026-07-02,
*Investigation — no decision*) documents a motivation to remove tldraw entirely:
it ships under a non-open-source licence, which is awkward for this AGPL-3.0
project. canvas-v2 (`@ensembleworks/canvas-sync`, `SyncClientPeer`) is that track.

**Does that make this upgrade wasted effort? No — but it does cap how much we
should invest.** The honest framing:

- tldraw-v1 is the **live** engine today; canvas-v2 is not yet the deployed
  engine. Work that keeps the live engine healthy still pays.
- The cost here is genuinely small: one real breaking change, behind an existing
  single-function boundary.
- It **reduces** tldraw coupling rather than deepening it — it avoids adding a
  direct `@tldraw/editor` dependency we'd otherwise take on.
- It is a patch-level move *within* v5, not a bet on tldraw's future.

If the replacement decision lands and canvas-v2 becomes the live engine sooner
than expected, the sunk cost is a few hours of type fixes. That is an acceptable
price for unblocking the lock affordance on the engine users are actually using.

## 9. Non-goals

- **Not** a move to 5.3.x — that line is canary/internal/next only; `latest` is
  5.2.5.
- **Not** the locked-shape affordance itself (separate spec — this only unblocks it).
- **Not** a canvas-v2 / tldraw-replacement decision.
- **Not** bundled with any other change. This branch does one thing.
