# Lab note: the very slow canvas box (2026-07-13)

**Host:** `ew-lsp-001`, Linux 6.12.95+deb13-amd64, 6 CPU, x86_64
**Investigator:** `pi` coding agent (terminal session `x4v45c8l`)
**Outcome:** the box was disk-I/O thrashing, not CPU-bound. Killed 15
memory-heavy, long-idle agent processes (1 `opencode` + 14 `claude`).
Load dropped from ~18 to ~2.7 and iowait from ~90% to 0%.

## Symptom

User reported the machine running very slowly.

First readings:

```
$ uptime
22:36:41 up 4 days, 13:27, load average: 17.96, 12.84, 7.09

$ free -h
               total   used    free   shared  buff/cache  available
Mem:            11Gi   7.5Gi   368Mi    40Mi   3.9Gi       3.9Gi
Swap:          2.0Gi   2.0Gi    28Ki

$ nproc
6
```

A load average of ~18 on a 6-core box looks at first glance like a CPU problem.
It is not. The `iostat` and `vmstat` traces told the real story:

```
$ iostat -xz 1 2
avg-cpu:  %user %nice %system %iowait %steal %idle
            7.54  0.00   7.21   85.25    0.00   0.00     ← 85–90% iowait, 0% idle

Device   r/s    rkB/s    r_await   %util
sdb     250.0  20084.0   1574.62   100.00   ← /home volume saturated
sda    1077.0  31700.0      0.27    11.20   ← root/swap volume paging in

$ vmstat 1 3
 r  b   swpd   free    buff  cache    si  so    bi    bo   in    cs  us sy id wa
 1 28 2096896 374172  59792 4056740   0  272 22652 352 9218 21290  4  5  0 90
 0 29 2097148 377720  59792 4057008   0   0 24400   0 9647 23559  5  4  0 91
```

Key signals:

- **`b` column (blocked threads) = 28–29.** Nearly thirty tasks stuck in
  uninterruptible-sleep (`D`) state, all waiting on the disk.
- **`wa` (iowait) ~90%, `id` (idle) 0%.** The CPUs are not computing; they are
  parked waiting on I/O.
- **`sdb` (the `/home` volume) at 100% util with ~1.57 s read latency.**
  Normal read latency for this kind of volume is single-digit milliseconds.
- **Swap exhausted.** `/swapfile` (on `sda`) is 2.0 GiB and 2.097 / 2.097 GiB
  used — essentially zero headroom. `si/so` shows active swap-out (`so=272`).

This is textbook **disk-I/O thrashing**: more resident memory is demanded than
RAM + swap can hold, so pages are constantly being faulted back in off a disk
that is already at 100% util. Everything queues behind the disk and the whole
box stalls.

```
Coaching request | 2026-07-09 11:22:43 | 2026-07-09 21:32:22 | 0.58 | 40108
```

Exactly **one** session ("Coaching request"), last touched
**2026-07-09 21:32** — four days ago. Cost frozen at $0.58, 40 108 output
tokens, never updated since. The DB file and WAL had not been written since
Jul 9 21:32 / 21:53 either.

### Conclusion for opencode

It was a 569 MB RSS + 181 MB swap parasite attached to a four-day-old,
never-revisited coaching chat. Its only live activity was the hourly
housekeeping loop and an inotify watcher, both of which kept faulting pages
back off the saturated disk. Killing it would lose nothing (the session
remains on disk in `opencode.db`, recoverable via `opencode --continue`) and
would free ~750 MB.

## Caveat about the shared canvas

Per `AGENTS.md` this box is a shared multiplayer canvas: teammates — humans
and other agents — share the room and watch live. The dormant `claude`
sessions therefore plausibly belonged to *teammates*, not this agent. Killing
them could interrupt an untouched-but-not-finished session. For that reason
the kill list was surfaced to the user for approval before action, and the
user selected the **aggressive** option (kill all 14 in the idle bucket).

## Actions taken

### 1. Kill `opencode` (PID 65883)

```
$ kill 65883          # SIGTERM
$ sleep 2; ps -p 65883 -o stat,rss,comm
   Dl+ 567340 opencode          ← still alive, stuck in D
$ kill -9 65883       # SIGKILL — lands next time it returns from disk
$ sleep 3; ps -p 65883
   opencode GONE
```

Note: a `D`-state process cannot act on `SIGTERM` until it next comes off the
disk queue. SIGKILL is also not delivered instantly while in `D`, but it is
honoured the moment the process returns from the blocking syscall. Both were
required here.

### 2. Aggressive sweep — kill the 14 idle `claude` processes

Targets (all 1–4 days old, ≤0.5% CPU):

```
705315 632426 131337 949340 384983 453015 793729
76190  776708 89579  497045 493546 496104 541313
```

Several were in `D` state, so `SIGKILL` was used directly (no point raising
SIGTERM on a process that cannot receive it until the disk frees — and the
disk frees only once some of these are gone).

```
$ for p in $TARGETS; do kill -9 "$p"; done
$ sleep 6
$ for p in $TARGETS; do ps -p "$p"; done          # no survivors
```

All 14 confirmed dead.

## Why this happened, and how to avoid it

The 11 GiB box was holding ~26 concurrent Claude-Code sessions plus an open
`opencode`, all at ~250–570 MB resident. Ram committed > RAM → swap filled
→ page-ins onto the same disk that other agents were also paging against →
disk at 100% util, multi-second read latency → every blocked task queues
behind the disk → load climbs far past `nproc` even though no CPU is busy.

Two structural takeaways for the canvas:

1. **Concurrency budget for agents should be memory-limited, not just
   process-count limited.** A room with 26 `claude` + `opencode` is ~25 `claude`
   past the RAM budget for this VM class. The scheduler/launcher (or whatever
   spawns fresh `claude`/`opencode` sessions per teammate) should refuse to
   start a new agent when `Mem.available` drops below a floor (e.g. ~3 GiB),
   or when swap utilisation crosses a threshold, rather than when a process
   count is hit.
2. **Idle agents should self-reap.** Sessions with no input for > N hours
   whose parent terminal has gone quiet are pure overhead on a shared box.
   Either the launcher should suspend/exit them after inactivity (keeping the
   session recoverable on disk, the way `opencode` does with its SQLite store)
   or the room should surface idle-agent memory as a stat teammates can clean
   up themselves.

The existing `docs/memory-resource-policy.md` is the right home for
(1)–(2); this note is the incident record.

## Repro commands (cheat-sheet)

```sh
# the headline triage
uptime; nproc; cat /proc/loadavg
free -h; cat /proc/swaps
iostat -xz 1 2
vmstat 1 3

# who's blocked on disk (D state)
ps -e -o pid,user,etime,pcpu,stat,rss,comm --no-headers | awk '$5 ~ /D/'
ps -e -o state --no-headers | grep -c ^D

# who's fattest
ps -eo pid,rss,comm --sort=-rss | head
ps -e -o comm --no-headers | sort | uniq -c | sort -rn

# forensic on a single suspect
tr '\0' ' ' < /proc/<pid>/cmdline; echo
readlink /proc/<pid>/cwd; readlink /proc/<pid>/exe
grep -E 'State|VmRSS|VmSwap|Threads' /proc/<pid>/status
cat /proc/<pid>/io
ls -l /proc/<pid>/fd 2>/dev/null | grep -v 'socket\|pipe\|/dev'
```

so the room could see what happened:

```

or run `ensembleworks auth login`
```

The `ensembleworks` CLI in this sandbox has no `ENSEMBLEWORKS_URL` configured,
so the status light and sticky could not be published. The work itself
(killing the thrashers) completed regardless; only the canvas visibility did
not. Worth wiring `ENSEMBLEWORKS_URL` (and a default room) into the agent's
environment so future incidents surface live on the canvas.
