Turning talk into INTAKE cards

People describe what they want out loud. This skill reads the room's voice transcript, finds the actual wants buried in it, and writes them onto the canvas as user-story cards — in the house format, with the right id, in the right place.

The skill owns the first box only. Deciding what is valuable, what is easy, and what ships next is a human judgement made by dragging cards rightwards — so the skill never promotes a card past INTAKE.

The card format

Four lines, exactly:

ID: JS<n>, User: <who asked>
As a <role>
I want to <capability>
So that <benefit>
RuleWhy
Ids are unique across all four frames Cards keep their id as they migrate rightwards, so the highest JS<n> may well be sitting in ROADMAP, not INTAKE.
User: is the person who asked Taken from the transcript's name field — not the agent, not the room. It's who to go back to with questions.
One want per card A single stretch of talk often holds two or three. Splitting them is what makes them prioritisable independently.
"So that" is a benefit, not a restatement If the So-that just paraphrases the I-want, the reason hasn't been found yet — keep reading the transcript.
Roles are concrete "product owner", "canvas A/V user", "terminal user" — drawn from how the speaker described themselves, never a generic "user".

How a run goes

  1. Read the room

    Pull the transcript tail and chain subsequent polls on the server's own now rather than the local clock. Every entry carries the speaker's name and the frame their cursor was in — that frame context is a strong hint about what they're talking about.

    ensembleworks scribe transcript --since $(( $(date +%s000) - 600000 ))

  2. Discard most of it

    Greetings, tangents, thinking-aloud and every "yeah" are not feature requests. A card is cut only when someone expresses a want or a pain with a discernible beneficiary. Silence is the correct output for most of any transcript.

  3. Read what's already on the canvas

    All four frames, for two reasons: to find the highest id in use, and to check the want isn't already carded. A duplicate is worse than a missed card — a near-match should be reported, not added.

    ensembleworks canvas frames
ensembleworks canvas frame INTAKE   # and PRIORITISER, ROADMAP, RELEASE NOTES

  4. Place the card

    Cards sit in a left-to-right row beneath the blank template, styled to match: yellow, solid fill, 384×202. The next x is computed from the live bounds of the last card in the row — never from a remembered coordinate, because humans drag these around constantly.

    ensembleworks canvas shape --op create --type geo --frame INTAKE \
  --geo rectangle --color yellow --fill solid \
  --w 384 --h 202 --x <next-x> --y 229 \
  --text 'ID: EW7, User: David
As a product owner
…'

  5. Say what you did

    A short sticky on the advice frame and an honest status light, so the room can see what was added without hunting for it.

One wrinkle worth knowing. Card geometry is read through /api/v2/canvas/frame, because the v1 frame reader deliberately returns only {id, type, text} for each shape — no coordinates, so there's no way to work out where the next card goes.

That endpoint is not the v2 canvas engine. It's a read-only view that converts the same legacy tldraw store on every request. The actual engine swap is /sync/v2/:roomId behind EW_CANVAS_SYNC, which this room does not run. All writes go through the ordinary v1 canvas shape endpoint.

Worked example

The conversation this skill was built from — David, cursor in the INTAKE frame, thinking aloud:

“I think one of the things that we want to ensure we have in the EW feedback and roadmap is some way of organizing the actual roadmap portion. And so I think this would be something that's kind of for the product owner, which is me, to have a good sense for the prioritization of what's coming up now, next and later. And for people who are using the EnsembleWorks tool, the students who are using it, for them to have a sense for where their feature request has been prioritized.”

The analytical move: one capability, but two roles with two different benefits. That's two cards, not one.

ID: EW7, User: David As a product owner I want the ROADMAP frame organised into now / next / later So that I can see what is coming up when
ID: EW8, User: David As an EnsembleWorks user I want to see where my feature request has been prioritised So that I know whether and when it will be picked up

Deliberately not carded from the same five minutes: a tangent about a toy penguin that plays a four-second loop, “can I come back to you in five minutes”, and a good deal of “yeah”. Knowing what to throw away is most of the job.
