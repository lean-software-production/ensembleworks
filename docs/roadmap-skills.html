Driving the roadmap board

Three small skills move a want along one pipeline on the EW: Feedback & Roadmap page — captured in INTAKE, prioritised into a ROADMAP zone, shipped into a release. Each hop is deliberate and human-triggered; nothing promotes itself.

The pipeline

INTAKEa want, as a cardroadmap-intake
→
ROADMAPNOW · NEXT · LATERroadmap-prioritise
→
RELEASE NOTESRELEASE: vX.Y.Zroadmap-release

One card = one want. It keeps its EW<n> id the whole way.

A card is created in INTAKE, prioritised into a zone, and — when it ships — moved into the matching release subframe with its PR link attached. The zone and the release call are human judgements; the skills just perform the hop cleanly when invoked.

PRIORITISER is retired. The old value/effort grid frame was renamed LATER and folded into ROADMAP. Prioritisation is now expressed purely as the NOW / NEXT / LATER zones — a three-stage pipeline, not four.

The three skills

SkillDoesMoves the card
roadmap-intake Reads the voice transcript, finds a real want, writes it as an EW<n> card in the house format. → INTAKE
roadmap-prioritise Reparents an existing card into a priority zone. The human names the zone; the skill never invents the priority. INTAKE → NOW / NEXT / LATER
roadmap-release Moves a shipped card into its release subframe (creating it if new) and stamps the implementation PR as a real link. → RELEASE NOTES / vX.Y.Z

The card format

Four lines. Once the want ships, the ID line also carries a real link to the PR that delivered it.

ID: EW2, User: David, PR: #60 As a canvas A/V user I want to have noise canceling So that noisy backgrounds don't ruin the meeting
  • EW<n> ids are unique across the whole page — a card keeps its id as it migrates, so allocate the next id by scanning everywhere, not just INTAKE.
  • User: is the person who asked (the transcript name), never the agent.
  • "So that" is the benefit, not a restatement of the want.

Two things the tooling makes you get right

ROADMAP and RELEASE NOTES are containers. Cards live in the leaf subframes (NOW/NEXT/LATER, RELEASE: vX.Y.Z), never as direct children — a v1 canvas frame ROADMAP read returns nothing. Scan the whole board through canvas-v2 document instead, and resolve frames by name every time, because humans reorganise the board constantly.

The PR link is richText, not markdown. [#60](url) renders as literal text, and the canvas shape --text path flattens through toRichText() and drops the link. The link must be written through --props as a raw richText doc whose #<n> node carries a link mark — exactly what the WYSIWYG editor writes when you hand-make a link.
